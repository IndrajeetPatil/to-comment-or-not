What Not to Comment

Every comment distracts from reading code‚Äîmake sure it‚Äôs worth the interruption.

Don‚Äôt use crutch comments

Crutch comments compensate for unclear code‚Äîfix the code instead.

‚ùå Comment compensates for bad naming

# Check if user can access premium features
def chk(u):
    return u.tier == 'gold' or u.tier == 'platinum'

‚úÖ Clear naming needs no comment

def has_premium_access(user):
    return user.tier in ['gold', 'platinum']

‚ùå Comment explains complex conditional

# Apply discount if eligible: active account,
# 3+ orders in last month, or loyalty member
if (user.is_active and user.orders_last_month >= 3
    or user.loyalty_status == 'member'):
    apply_discount(user)

‚úÖ Well-named function obviates comment

def is_discount_eligible(user):
    frequent_buyer = (user.is_active and
                     user.orders_last_month >= 3)
    loyalty_member = user.loyalty_status == 'member'
    return frequent_buyer or loyalty_member

if is_discount_eligible(user):
    apply_discount(user)

Don‚Äôt state the obvious

‚ùå Bad

# increment counter
counter += 1

‚úÖ Good

# Reset retry count after successful connection
counter = 0

Don‚Äôt repeat variable names

‚ùå Bad

# user name variable
user_name = "john"

‚úÖ Good

# Store normalised username for database
user_name = normalize(input)

Don‚Äôt explain language syntax

‚ùå Bad

# if statement checks condition
if user.is_active():
    process(user)

‚úÖ Good

# Only process active users per business rule
if user.is_active():
    process(user)

Don‚Äôt retain dead code as comments

‚ùå Bad

# old_function()
# legacy_code = True

‚úÖ Good

new_function()

Don‚Äôt write vague action comments

‚ùå Bad

# TODO: optimize this
process_data()

‚úÖ Good

# TODO: Switch to binary search (issue #1234)
process_data()

Don‚Äôt confuse docs with comments

‚ùå Bad

"""Implementation uses binary search
for O(log n) complexity"""

‚úÖ Good

"""Find user by ID."""
# Use binary search for O(log n) lookup

What to Comment

Whatever helps the reader understand the code more easily: the what, the why, and/or the how.

Explain the Thought Process

‚ùå Comments reiterate code

# Timeout value is 30 seconds
timeout = 30

# Connect to database
db.connect(retry=True)

# as per spec
if len(data) == 0:
    return None

‚úÖ Comments explain rationale

# Timeout to handle slow network conditions
# in international deployments
timeout = 30

# Retry enabled to handle observed
# transient network failures
db.connect(retry=True)

# Return None for empty datasets per
# API specification v2.1 section 4.3
if len(data) == 0:
    return None

Tip

Good comments explain business logic, design decisions, and non-obvious requirements.

Document Non-Obvious Logic

Complex logic benefits from explanation‚Äîeven when developers could eventually figure it out.

‚ùå Cryptic regex

pattern = r'^(?=.*[A-Z])(?=.*\d)(?=.*[@$!%*?&])[A-Za-z\d@$!%*?&]{8,}$'
if re.match(pattern, password):
    return True

‚úÖ Regex requirements explained

# Password must have: uppercase letter, digit,
# special char (@$!%*?&), min 8 characters
pattern = r'^(?=.*[A-Z])(?=.*\d)(?=.*[@$!%*?&])[A-Za-z\d@$!%*?&]{8,}$'
if re.match(pattern, password):
    return True

‚ùå Unexplained bit manipulation

def is_power_of_two(n):
    return n > 0 and (n & (n - 1)) == 0

‚úÖ Logic clarified

def is_power_of_two(n):
    # Powers of 2 have single bit set;
    # n & (n-1) clears lowest bit, yields 0
    return n > 0 and (n & (n - 1)) == 0

Tip

Complex logic deserves explanation‚Äîclarity trumps brevity.

Explain Unidiomatic Code

When you intentionally deviate from language conventions, explain why the idiomatic approach doesn‚Äôt work.

‚ùå No explanation for unusual approach

# Process items
result = []
for item in items:
    if item > 0:
        result.append(item * 2)

‚úÖ Explains deviation from idiom

# Use explicit loop instead of list comprehension
# to avoid memory issues with large datasets (10M+ items)
result = []
for item in items:
    if item > 0:
        result.append(item * 2)

‚ùå Unidiomatic without context

file = open('data.txt', 'r')
content = file.read()
file.close()

‚úÖ Justifies non-Pythonic code

# Cannot use context manager: file handle must remain
# open for async callback (library limitation)
file = open('data.txt', 'r')
content = file.read()
file.close()

Tip

Unidiomatic code raises questions‚Äîanswer them preemptively with comments.

Highlight Known Flaws

It‚Äôs acceptable to document known issues, limitations, or future improvements using action comments.

‚ùå Vague action comment

# TODO: fix this
def process_data(items):
    return [x * 2 for x in items]

‚úÖ Specific with tracking

# TODO: Add validation for empty list (issue #847)
def process_data(items):
    return [x * 2 for x in items]

Common Action Comment Tags

Tag Purpose Example
TODO Planned improvement or missing feature # TODO: Add caching (issue #123)
FIXME Known bug that needs fixing # FIXME: Fails on negative input (#456)
HACK Temporary workaround for a problem # HACK: API bug workaround (ticket #789)
NOTE Important clarification or caveat # NOTE: Must run before init()
OPTIMIZE Performance improvement opportunity # OPTIMIZE: Use binary search (#234)

Tip: Use the Better Comments VS Code extension to highlight these tags in your editor.

How to Comment

Keep them precise and compact.

Use information-dense words

‚ùå Verbose

# Store function results based on inputs
# to avoid recomputing same values
@save_results
def calculate(x, y):
    return x ** y

‚úÖ Uses technical term

# Memoize expensive computation
@save_results
def calculate(x, y):
    return x ** y

Avoid pronouns

‚ùå Ambiguous pronouns

# Sync local cache with remote database
# and invalidate it
sync(local_cache, remote_db)
invalidate()

‚úÖ Specific nouns

# Sync local cache with remote database
# and invalidate local cache
sync(local_cache, remote_db)
invalidate()

Explain intent, not mechanics

‚ùå Lower-level details

# Loop through array and add to sum
total = sum(prices)

‚úÖ Higher-level intent

# Calculate order total for tax computation
total = sum(prices)

Be accurate

‚ùå Misleading comment

# Calculate average
return median(values)

‚úÖ Accurate description

# Calculate median for anomaly detection
return median(values)

Keep comments updated

‚ùå Outdated comment

# TODO: fix this hack (3 years old)
legacy_workaround()

‚úÖ Current and relevant

# Use new API endpoint per v2.0 migration
refactored_solution()

Use specific references

‚ùå Vague reference

# See the function above
apply_discount()

‚úÖ Clear reference

# Rate logic in pricing.calculate_discount()
apply_discount()

Maintain professional tone

‚ùå Unprofessional

# This code is a mess
process_legacy_data()

‚úÖ Professional

# Complex legacy integration, needs refactoring
process_legacy_data()

Avoid inside jokes

‚ùå Unclear to outsiders

# Here be dragons
process_transaction()

‚úÖ Clear explanation

# Handles concurrent writes with optimistic locking
process_transaction()

Stay objective

‚ùå Emotional/subjective

# Stupid requirement from management
validate_input()

‚úÖ Factual

# Business requirement: process within 24h
validate_input()

Tools & Techniques

Tool Limitations

What tools CAN do:

  • Check comment formatting and style
  • Flag TODO/FIXME comments
  • Detect missing documentation
  • Generate basic API documentation

What they CANNOT do:

  • Understand if comments explain the ‚Äúwhy‚Äù
  • Assess comment usefulness and clarity
  • Determine if business context is missing
  • Evaluate comment accuracy after code changes
  • Judge whether comments add value

The fundamental limitation

Tools can enforce format but not value. Good commenting requires human judgment about what information is helpful.

AI as an Ally

Why AI tools can help:

  • Analyze code complexity and suggest documentation
  • Identify business logic that needs explanation
  • Check comment clarity and helpfulness
  • Generate initial documentation drafts

Symbiotic Commenting

Write initial comments yourself. Then, ask AI tools to identify gaps, suggest improvements, or validate clarity.

Cross-LLM Review

Use a different LLM to evaluate comments than the one used to write the code.

Code Review Benefits

Code review transforms commenting from reactive to deliberate!

Benefits: Context and Understanding

  • Well-written comments make complex code comprehensible.
  • Writing thoughtful comments forces you to articulate your reasoning and design decisions, improving code quality.
  • Good comments preserve business knowledge and domain expertise for future developers.
  • Strategic commenting reduces onboarding time for new team members.
  • Consistent commenting practices reduce cognitive overload and make maintenance safer.


Invest time in thoughtful comments early‚Äîpractice makes perfect.
